# 2주차 Finance Agent 확장 계획

## 목표
1주차 엔진(스키마/계약/정책 로드/비용 분해)을 **정책 고정 + 버전 + 무결성 검증 + 추천 제약 + 시나리오 회귀테스트**까지 포함하도록 확장.

## 설계 원칙 (준수 사항)
- 상수/테이블 하드코딩 금지 → policy 파일에서만 로드
- Policy 무결성: manifest + sha256 검증, 불일치 시 실패
- 결과 감사추적: policy_version, policy_hash, applied_profile, applied_regulation_weights 포함
- Action 추천: action_catalog에 있는 action_id만 허용
- 제약: recommendation_constraints.json 룰로만 판단 (코드 하드코딩 금지)
- 회귀 테스트: tests/scenarios 6개 JSON + pytest

## 변경 파일 목록 및 패치 계획 (작은 커밋 단위)

### 커밋 1: Policy 디렉터리 및 v1.0.0 JSON 파일
- **추가**
  - `policy/policy_manifest.json` (policy_version, files: path → sha256)
  - `policy/v1.0.0/impact_table.json`
  - `policy/v1.0.0/likelihood_table.json`
  - `policy/v1.0.0/regulation_weights.json`
  - `policy/v1.0.0/profile_weights.json`
  - `policy/v1.0.0/action_catalog.json`
  - `policy/v1.0.0/recommendation_constraints.json`
  - `policy/v1.0.0/policy_meta.json` (pricing_table, combine_mode, approved_by 등)
- **유지**: `policy/policy.v1.0.json` (기존 엔진 호환용, 점진적 마이그레이션 후 제거 가능)

### 커밋 2: Policy 로더 확장 + manifest 해시 검증
- **수정**: `src/policy_loader.py`
  - `load_policy(version)` → version이 `v1.0.0` 형식이면 `policy/<version>/` 디렉터리에서 번들 로드
  - `load_manifest()` / `validate_manifest_hashes(manifest_path)` → 각 파일 sha256 검증
  - `compute_manifest_hash(manifest_json)` → 결과에 기록할 manifest 해시
  - 반환: 단일 policy 번들 dict (impact_table, likelihood_table, regulation_weights, profile_weights, action_catalog, recommendation_constraints, policy_meta, pricing_table)
- **추가**: `scripts/build_manifest.py` → policy/v1.0.0/*.json 읽어서 sha256 채운 policy_manifest.json 생성

### 커밋 3: 하드코딩 제거 및 점수/추천 policy 기반 리팩터
- **수정**: `src/risk_model.py`  
  - impact/likelihood/regulation_weight → 인자로 받은 policy 번들에서 조회 (policy.py 사용 제거)
- **수정**: `src/scoring_engine.py`  
  - action_catalog, profile_weights → policy 번들에서 조회 (actions.py, policy.py 사용 제거)
  - recommendation_constraints 적용: 룰 평가 후 forbid된 action 제외 후 스코어링
- **삭제 또는 비우기**: `src/policy.py` → policy에서 로드하므로 상수 제거, 또는 deprecated로 두고 policy_loader에서 채운 dict만 사용
- **삭제 또는 비우기**: `src/actions.py` → action_catalog는 policy에서 로드, get_actions(policy_bundle) 형태로 변경

### 커밋 4: 요청/결과 스키마 및 contract 확장
- **수정**: `schemas/finance_request.schema.json`  
  - assumptions에 `scenario_class` (S1|S2|S3), `profile` (Default|MissionCritical|PII_Heavy|FraudSensitive), `regulation_context` (framework, weight_profile) 추가 또는 기존 service_tier/org_profile 유지하고 의미 정합
- **수정**: `schemas/finance_result.schema.json`  
  - `policy_hash`, `applied_profile`, `applied_regulation_weights`, `recommendation` (action_id), `top_actions` (선택), driver_breakdown에 `why` (선택)
- **수정**: `src/contract.py`  
  - 허용 enum은 policy에서 오도록 하거나, 스키마 enum과 동기화 (최소 변경 시 기존 service_tier/org_profile 유지, profile/regulation_context 추가 시 policy의 profile_weights 키 등과 매핑)

### 커밋 5: Engine 결과에 policy_hash, applied_*, recommendation 반영
- **수정**: `src/engine.py`
  - load_policy → 새 policy_loader 사용 (번들 로드 + manifest 검증)
  - 비용 계산 후, policy 번들로 risk/스코어/추천 계산 (scoring_engine 호출 시 policy 전달)
  - 결과에 policy_hash, applied_profile, applied_regulation_weights, recommendation (catalog 내 action_id만) 포함
  - driver_breakdown에 why 필드 추가 (숫자 기반 설명)

### 커밋 6: 시나리오 6개 + pytest
- **추가**: `tests/scenarios/` 아래 JSON 6개
  - S1 + high severity → ISOLATE_INSTANCE 추천 금지
  - S1 + high + allow 조건 → 격리 허용
  - S3 + high → 격리 추천 가능
  - S3 + high + audit weight 높음 → 격리 + LOG_PRESERVE 상위
  - regulation encryption weight 낮음 vs 높음 → ROTATE_KMS_KEY rank 변화
  - regulation access_control weight 낮음 vs 높음 → DISABLE_ACCESS_KEY/MFA rank up
- **추가**: `tests/test_scenarios.py`  
  - 시나리오 JSON 로드 → finance_run 호출 → 필수 포함/미포함·순위 변화 assert

## 기존 동작 유지
- `policy_version: "v1.0"` 요청 시 기존 policy.v1.0.json 로드 경로 유지 가능 (호환 레이어) 또는 v1.0 → v1.0.0 매핑
- 같은 입력 + 같은 policy_version → 항상 같은 결과 (재현성)
